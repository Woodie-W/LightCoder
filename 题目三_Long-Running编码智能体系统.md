# 题目三：Long-Running 编码智能体系统

**资源要求**：可通过官方 API 或本地部署方式调用大语言模型（例如 DeepSeek-V4-Flash、Qwen3.7 等）。本题为 **Training-free 设定**，不涉及模型训练。

## 一、任务要求

请实现一个可通过命令行运行的最小化编码智能体系统，使其能够在尽量少的人工干预下**持续自主运行数小时量级的时长**，并持续产出可验证的进展。

编码智能体的基本工作方式，是由模型进行决策，调用工具与本地环境交互，再将执行结果返回模型以驱动下一轮决策。当任务规模明显超出单个上下文窗口的容量时，这一循环难以直接维持。首先，上下文窗口容量有限，模型输出的连贯性也会随上下文长度增长而下降。其次，模型的规划能力不足，倾向于一次性完成全部工作，常常留下未完成且难以交接的中间结果。最后，模型难以可靠地评判自身输出，可能将实际未完成的工作判定为已完成。

**本题关注的核心问题是**：系统应当如何管理任务状态并规划当前工作；如何在上下文容量约束下维持工作的连贯性；如何可靠判定任务是否真正完成；以及哪些经验与状态值得跨会话持久化。系统核心流程需要**自主实现，不允许直接使用现成智能体框架**。系统应包含以下基本组成：

### 1. 基础智能体循环

系统接收任务，调用模型进行决策，并通过必要的工具与本地环境交互。工具能力保持最小化即可，例如：

- **bash**：执行命令行指令
- **read**：读取文件内容
- **write**：写入或修改文件

### 2. 长时程运行能力

系统需要能够承接规模明显超出单个上下文窗口的编码任务并持续推进，该能力由三项机制构成：

- **任务状态管理**：显式管理任务分解与执行进度，说明规划粒度如何确定，以及系统如何决定当前应执行的工作与整体终止时机。
- **上下文管理**：在容量约束下维持工作连贯性，方案不限（如历史压缩、会话切换与交接等），说明跨越上下文边界时哪些信息必须传递、以何种形式传递。
- **自我验证**：判定任务是否真正完成，说明完成标准如何定义、依赖模型评判自身输出的可靠性如何、引入独立于生成过程的检验方式的收益与代价。

### 3. Skill 与 Memory 机制

系统以文件系统为载体实现跨会话的经验与状态持久化：

- **Skill**：指可复用的任务经验、操作流程、提示模板或程序片段
- **Memory**：指任务进度、已确认的事实、架构决策等运行状态

需要说明：

- 哪些信息应写入持久化文件、哪些应在需要时现场读取
- 持久化内容如何被后续运行检索与利用
- 分析错误记录、冗余记录和记录污染等问题

### 4. 实验分析

设计至少一个规模超出单个上下文窗口的长程编码任务作为评测载体，报告自主运行情况（如运行时长、上下文压缩或会话切换次数、任务完成程度及验收依据），并至少给出一组关闭某项机制的对照实验。

> 本题不要求任务完全完成，若观察到重复执行已完成工作、过早判定任务完成等现象，能基于运行轨迹给出准确归因，同样是重要的考核内容。

## 二、设计说明文档需覆盖的内容

- 长时程运行能力（任务状态管理、上下文管理、自我验证）
